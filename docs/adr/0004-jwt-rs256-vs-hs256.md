# ADR 0004 — JWT RS256 (asymétrique) vs HS256 (symétrique)

## Status

Accepted (2026-04-18)

## Context

NEXYA expose une API mobile + future web app. Les tokens JWT signent
les sessions (access TTL 15 min + refresh TTL 30 j).

Choix entre :
1. **HS256** (HMAC SHA-256) — secret partagé symétrique
2. **RS256** (RSA SHA-256) — clé privée signe / clé publique vérifie
3. **ES256** (ECDSA P-256) — variante elliptique RS256

## Decision

**RS256 asymétrique**.

## Consequences

### Positives

- **Clé publique distribuable** sans risque
  - V2 : si NEXYA expose un endpoint `/auth/jwks` (JSON Web Key Set),
    les clients (mobile app, futur SDK partenaires) peuvent vérifier
    le token localement sans appel backend
  - Permet auth inter-services future (un microservice NEXYA peut
    vérifier sans avoir le secret)
- **Rotation clé privée** sans impacter les vérifieurs (juste push de
  la nouvelle clé publique)
- **Standard industry** : OpenID Connect impose RS256 minimum
- **Audit-friendly** : les vérifieurs n'ont pas le pouvoir de signer
- **Pas de fuite par log** : la clé privée vit côté backend uniquement,
  jamais dans un log/Sentry breadcrumb (scrubber ne devrait jamais la
  voir)

### Négatives

- **Performance** : ~10× plus lent que HS256 pour signature
  (acceptable : signature 1× par login = ~1 ms RSA-2048, vérification
  ~0.1 ms)
- **Génération clés** : `openssl genrsa -out private.pem 2048` puis
  extraction publique (vs `openssl rand -hex 32` pour HS256)
- **Taille clé** : ~1.6 KB private RSA-2048 vs 32 bytes HS256

### Mitigations

- **Setup README** documenté avec commande `openssl` exacte
- **Production safety guard** refuse au boot si `JWT_PRIVATE_KEY` ou
  `JWT_PUBLIC_KEY` vide
- **Performance acceptable** — signature 1× par login, pas hot path
  chat
- **Migration ES256** envisageable V2 si perf critique (~5× plus rapide
  que RS256 ECDSA P-256)

## Implémentation

### Génération clés (1 fois)

```bash
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem
```

### Stockage

```bash
# .env
JWT_PRIVATE_KEY=-----BEGIN PRIVATE KEY-----
<contenu private.pem>
-----END PRIVATE KEY-----

JWT_PUBLIC_KEY=-----BEGIN PUBLIC KEY-----
<contenu public.pem>
-----END PUBLIC KEY-----
```

### Encode access token

```python
# app/core/auth/jwt.py
def create_access_token(user_id: UUID) -> str:
    payload = {
        "sub": str(user_id),
        "purpose": "access",
        "iat": now,
        "exp": now + timedelta(minutes=15),
        "jti": str(uuid4()),
    }
    return jwt.encode(payload, settings.jwt_private_key, algorithm="RS256")


def decode_access_token(token: str) -> dict:
    return jwt.decode(token, settings.jwt_public_key, algorithms=["RS256"])
```

### Refresh token rotation

- Hash SHA-256 stocké en DB (`refresh_tokens.token_hash`)
- Rotation : à chaque usage, ancien hash révoqué + nouveau émis
- TTL 30 j

### Tokens dérivés

NEXYA utilise RS256 aussi pour :
- **Reset password token** (TTL 15 min, purpose=`password_reset`,
  fingerprint hash du password actuel)
- **Unsubscribe token** (TTL 365 j, purpose=`email_unsubscribe`,
  catégorie whitelist)

Le `purpose` claim empêche la confusion entre tokens — le validateur
vérifie strict `purpose=expected_value`.

## Alternatives considérées

### HS256

**Pour** : simplicité, performance, secret unique.

**Contre** :
- **Symétrique** : le secret pour vérifier = le secret pour signer →
  fuite par log/breadcrumb très grave
- Pas distributable (mobile app ne peut pas vérifier localement)
- Rotation = downtime (tous les tokens deviennent invalides)
- Anti-pattern industry pour OIDC

### ES256

**Pour** : performance ECDSA, taille clé compacte.

**Contre** :
- Adoption plus tardive — quelques librairies tierces ont des bugs
- RS256 mature + écosystème énorme
- Migration V2 possible si benchmark perf le justifie

## Notes

V2 — envisager :
- **Endpoint JWKS** `/auth/.well-known/jwks.json` pour les clients
  qui veulent vérifier sans backend
- **Rotation automatique** clé privée tous les 6 mois (cron + grace
  period 30 j)
- **MFA TOTP** layer additionnel par-dessus (optionnel Pro)
